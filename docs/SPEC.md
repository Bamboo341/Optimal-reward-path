## 1. システム概要

OpenStreetMap（OSM）の道路ネットワークに対し、
(A) 道路エッジへ手動で報酬（ポイント）を設定し、
(B) 出発点・到着点・距離上限を指定して**取得報酬を最大化する経路**を探索・表示する、
ローカル動作のツールを作成する。

- 利用者: 開発依頼者本人のみ（シングルユーザー、認証不要）
- 動作環境: ローカルPC（Windows想定だがOS非依存で実装すること）
- UI: Streamlit によるブラウザUI（`streamlit run app.py` で起動）

## 2. 技術スタック（固定）

| 項目 | 採用技術 |
|---|---|
| 言語 | Python 3.11+ |
| グラフ取得/処理 | OSMnx 2.x / NetworkX |
| UI | Streamlit + streamlit-folium |
| 地図描画 | folium (Leaflet) |
| データ保存 | JSONファイル（`data/` 配下） |
| テスト | pytest |

## 3. 対象地域とグラフ仕様

- 既定対象: `"Shimogyo-ku, Kyoto, Japan"`（京都市下京区）
- `config.yaml` の `place` を変更するだけで京都市全域等に切替可能であること
- 取得: `ox.graph_from_place(place, network_type="all", simplify=True)`
- **取得後、無向 MultiGraph に変換して以後すべて無向で扱う**（徒歩前提のため一方通行は無視する。設計判断）
- 初回取得後 `data/graph_{place_slug}.graphml` にキャッシュし、2回目以降はローカル読込
- エッジ距離: OSMnxの `length` 属性（メートル）をコストとして使用

### エッジの同定

無向グラフのエッジは `(u, v, key)` で同定する。保存時は `u < v` に正規化する。

## 4. 問題の形式的定義

無向グラフ G=(V, E)、各エッジ e に距離 l(e) > 0。
報酬エッジ集合 R ⊆ E、各 r ∈ R に報酬 p(r) > 0。
出発ノード s、到着ノード t、距離上限 L が与えられる。

**目的**: s から t への経路 W で、総距離 Σl(e) ≤ L を満たし、
W が1回以上通過した報酬エッジの報酬合計 Σ_{r ∈ R∩W} p(r) を最大化する。

- **同一報酬エッジを複数回通っても報酬は1回のみ加算**
- 探索モードは2種類:
  - **モードR（再訪許可）**: W は歩道（walk）。同一エッジ・ノードの再訪自由
  - **モードS（再訪禁止）**: W は小道（trail）。**同一エッジを2度通ることを禁止**（ノード再訪は許可。設計判断: 報酬がエッジ単位のため）
- 厳密解は不要。**近似解でよいが、探索は100秒以内に完了すること**

この問題は Arc Orienteering Problem（NP困難）である。

## 5. アルゴリズム仕様

### 5.1 中核となる設計方針: メタグラフ化

報酬エッジは手動設定のため少数（想定: 5〜50本）である。この前提を利用する。

1. メタノード集合 = { s, t } ∪ { 全報酬エッジの両端点 }
2. 各メタノードから元グラフ上で Dijkstra を実行し、メタノード間最短距離 d(x, y) と最短経路を前計算する
3. 解 = 「訪れる報酬エッジの順列＋各エッジの通過方向」。
   報酬エッジ r_i = (a_i, b_i) を方向 o で通るとき、入口ノード in(r_i,o)、出口ノード out(r_i,o) が定まる。
   総距離 = d(s, in(r₁)) + l(r₁) + d(out(r₁), in(r₂)) + … + l(r_k) + d(out(r_k), t)

モードR（walk）ではこの定式化は損失がない（最短接続の連結で必ず実現可能）。

### 5.2 モードR の探索

**|R| ≤ N_exact（既定 14、config可変）の場合: ビットマスクDPによる厳密解**

- 状態: `dp[mask][i][o]` = 報酬エッジ集合 mask を回収済みで、最後にエッジ i を方向 o で通過し終えた時点の最小総距離
- 遷移: 未回収エッジ j・方向 o' へ `dp[mask][i][o] + d(out(i,o), in(j,o')) + l(j)`
- 解: 全状態について `dp + d(out, t) ≤ L` を満たすものから報酬最大を選ぶ
- 経路復元: 前計算したDijkstra経路を連結して実ノード列を得る

**|R| > N_exact の場合: ヒューリスティック（時間上限つき）**

- 初期解: 貪欲挿入（報酬/追加距離 比が最大のエッジを距離制約内で逐次挿入）
- 改善: ローカルサーチ（挿入・削除・入替・2-opt・方向反転）をランダムリスタート付きで反復
- `time_limit_sec`（既定 90）到達で打ち切り、その時点の最良解を返す

**報酬の再集計**: 復元した実経路が接続部で偶然別の報酬エッジを通過した場合も報酬に加算する（実経路を走査して最終報酬を再計算する）。

### 5.3 モードS の探索（ヒューリスティックでよい）

1. モードRと同じ手順で報酬エッジ順列の候補を得る
2. 実経路の構築時、**使用済みエッジに大きなペナルティ（例: length×10⁶）を課した Dijkstra** を接続区間ごとに逐次実行し、エッジ重複を回避する
3. どうしても重複が必要な接続が生じた場合、その報酬エッジをスキップして再試行する（修復）
4. 完成した経路が trail 条件と距離制約を満たすことを検証してから解として採用する

モードSの解は近似であり最適性保証は不要。ただし**返す解は必ず制約を満たす実行可能解であること**。

### 5.4 複数候補の出力

- 上位 K 件（既定 K=3、UI指定可）を返す
- **回収する報酬エッジ集合（mask）が互いに異なる**解のみを候補として採用する（同一集合で経路だけ違うものは1件に統合）
- 各候補: 経路ノード列、総距離、総報酬、回収した報酬エッジ一覧

## 6. データ仕様

### 6.1 報酬データ `data/rewards_{place_slug}.json`

```json
{
  "version": 1,
  "place": "Shimogyo-ku, Kyoto, Japan",
  "rewards": [
    {
      "u": 1234567,
      "v": 2345678,
      "key": 0,
      "reward": 50,
      "memo": "四条通の一部",
      "road_name": "四条通"
    }
  ]
}
```

- `u < v` に正規化して保存
- 読込時、グラフに存在しないエッジは警告を出してスキップ（グラフ更新への耐性）

### 6.2 探索結果エクスポート（任意機能）

候補経路を GeoJSON（LineString + properties: reward, length）として `output/` に保存できること。

## 7. UI仕様

Streamlit のサイドバー等で2モードを切り替える。

### 7.1 報酬設定画面

- folium 地図を表示。設定済み報酬エッジは赤色ハイライト＋報酬値のツールチップ表示
- 操作フロー:
  1. 地図上の任意点をクリック
  2. 最近傍エッジ（`ox.distance.nearest_edges`）を青色でプレビュー表示、道路名を表示
  3. サイドバーで報酬値（正整数）とメモを入力し「設定」ボタンで保存
- 設定済みエッジの一覧表示、個別削除、報酬値の変更が可能であること
- 保存は即時にJSONへ書き込む

### 7.2 経路探索画面

- 入力:
  - 出発点・到着点: 地図クリック → 最近傍ノードにスナップ（マーカー表示）
  - 距離上限 L（メートル、数値入力）
  - モード選択: 再訪許可 / 再訪禁止
  - 候補数 K（1〜5、既定3）
- 「探索実行」ボタンで実行。実行中はスピナー表示
- 出力:
  - 候補経路を色分けして地図に重畳表示（レイヤーで表示/非表示切替可能）
  - 候補ごとの表: 総報酬 / 総距離 / 距離上限に対する使用率 / 回収報酬エッジ数
  - 実行可能解が存在しない場合（s-t最短距離 > L）はその旨をエラー表示

## 8. 非機能要件

- 探索時間: 100秒以内（内部の time_limit は90秒に設定し余裕を持たせる）
- グラフ読込は `st.cache_resource` でキャッシュし、UI操作のたびに再読込しないこと
- 下京区規模（目安: ノード数千〜1万）で全UI操作が快適に動作すること
- 京都市全域でも動作すること（描画は重くなってよいが、探索は5.1の方式によりスケールする）
- 地図描画負荷対策: 全エッジのGeoJSON描画はせず、背景はfoliumタイル＋報酬エッジと経路のみ描画する

### Phase 0: 基盤
- グラフ取得・graphmlキャッシュ・無向化・configの実装

### Phase 1: 報酬設定機能
- 7.1のUI一式とJSON永続化
- 受入: クリック→エッジ特定→報酬設定→再起動後も保持、の一連が動作する

### Phase 2: モードR探索（コア）
- メタグラフ構築・ビットマスクDP・経路復元・地図表示（候補1件）
- 受入: tests の合成グラフで既知最適解と一致。下京区・報酬10本・L=5kmで10秒以内に解が返る

### Phase 3: 拡張探索
- ヒューリスティック（|R|大）、モードS、複数候補K件表示
- 受入: 報酬30本でも90秒以内に解が返る。モードSの解にエッジ重複がないことを検証コードで確認

### Phase 4: 仕上げ
- GeoJSONエクスポート、エラーハンドリング、README整備
- 受入: pytest全通過、README手順のみで新規環境で起動できる

## 11. テスト仕様

- **合成グリッドグラフ**（例: 5×5、全エッジ長100m）に報酬を数本配置し、最適解が既知のケースをpytestで検証する（DPの正当性確認）
- 実行可能性テスト: 返却経路が (a) s-t を接続 (b) 総距離 ≤ L (c) モードSでエッジ重複なし、を常に満たす
- 報酬1回加算テスト: 同一報酬エッジを2回通る歩道で報酬が重複加算されないこと

## 12. 既知のリスクと対処方針

| リスク | 対処 |
|---|---|
| OSMnx初回取得にネットワーク必要 | graphmlキャッシュ必須。取得失敗時は明示エラー |
| PythonでのビットマスクDPが遅い | N_exactを14既定とし、超過時はヒューリスティックへ自動フォールバック |
| streamlit-foliumのクリックイベント仕様変更 | st_foliumの返却値 `last_clicked` を使用。バージョンをrequirements.txtで固定 |
| 京都市全域での描画負荷 | 報酬エッジと経路のみ描画する方針で回避 |
